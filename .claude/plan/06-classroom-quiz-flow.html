● Agentic MCQs Auto-Generation Workflow · System Architecture

One reading in.
A portal-ready quiz out.

A self-correcting, human-supervised pipeline behind three workflows — a Classroom Quiz (one session split into Set 1 / Set 2 + typed variants), an MCQ Practice set (one whole-session pool, all content types, no variants), and a Module Quiz (several sessions merged into one assessment, consolidated outcomes + typed variants). It writes questions, scores them against a single rubric, pauses for a human at each gate, and learns from every rejection.

2 The three human gates

The agent does the work — the human makes the calls

Each gate pauses the run and shows the work in a Set 1 / Set 2 two-column review, with exactly one "Action needed" banner at a time, so it's never ambiguous whether you must act.

1 Generation gate

The freshly generated questions, each with its concept-coverage check. You accept, improve, or delete. Deleting asks why — and that reason feeds learning.

2 Rubric gate

Each accepted question scored green / red against the 25 per-question checkpoints, with a plain-language "what's wrong + a suggested fix". You approve · improve · reject.

3 Variants gate

Every approved question with its its typed variants beneath it, each scored. You approve · improve · reject before the portal files are written.

🔁 Every "no" — and every "improve" — teaches. A reason is required to reject or delete, and Improve (you type feedback → the agent rewrites that question → you confirm) feeds the same loop. The feedback is distilled into a generic authoring rule and, on recurrence, sharpens the rubric checkpoint itself (see §5). The system gets better every time you correct it — not only when you reject.
3 One evaluator

A single self-evolving rubric — and Python does the math

No gold dataset, no spreadsheet. Every question is judged only against the rubric, which lives in memory and is refined over time. The LLM returns only which checkpoints failed; the score is computed in deterministic Python — so the model never grades itself. Each checkpoint is scoped: the code-quality checkpoints apply only to code questions, so a text question's bar is unchanged.

25 per-question checkpoints · 4 categories · all required

Question Text 9/9

  • Clarity — understandable in one read
  • No platform/tool reference unless needed
  • No analogy as the question's basis
  • Tests understanding, not syntax recall
  • No unknown jargon

Options 8/8

  • Uniform length (~50–70 chars)
  • No partial correctness in wrong options
  • Distractors meaningfully plausible
  • Parallel option format
  • No "None"/"All" misuse

Answer Validity 3/3

  • Definitively correct — not "most correct"
  • Length doesn't give it away
  • Derivable from the session only

Pedagogy 5/5

  • Right Bloom level (apply/analyze)
  • No subjective questions
  • No answer-by-elimination
  • Grounded real-world scenarios
  • Keyword highlighting

Every checkpoint in every category must be met — a single miss (in any category) bands the question RED.

+ 5 code-scoped checkpoints · code questions only (Feature 7) · all required

Code Question Quality up to 5

  • Code shown & complete — no …, snippet not in the question text
  • Code must not reveal the answer — no give-away comment or printed result
  • OUTPUT matches execution — exact, for textual & fill-in-the-blank
  • FIB: one correct InlineBlank at the tested token
  • Error/fix tied to the learning outcome — minimal, LO-explainable

Applicability is per question type: a text MCQ stays at 25 per-question points; a code-analysis MCQ adds 3 (→ 28), a fill-in-the-blank coding question adds 4 (→ 29). New checkpoints ride the same reject→distill→promote loop, so the rubric keeps learning for code too.

5 per-set checkpoints · the whole set · all required

Type & Format + Redundancy 4/4 · 1/1

  • ≥3 distinct question types in a set
  • Both True & False variants present
  • Code shown where the question needs it
  • Match-the-following mappings unambiguous
  • No redundancy — no two questions test the same thing (1.4)

Banding — a binary gate

🟢 GREEN — pass

Every applicable checkpoint is met.

🔴 RED — fail

Any unmet checkpoint → auto-optimize loop fires; still red after the cap → human gate.

No minimums, no amber — an unambiguous pass/fail. The per-category counts below show only what failed.

Same rubric, two scopes: per_question inside the scoring loop, per_set once over the finished variant set before export.

One real question, followed through five cases

Q001 · introduction_to_langchain · set 1 — “When working directly with LLM providers without a framework, which of the following is a manual repetitive task developers must perform?”
OPTION 1 — Automatically scaling infrastructure  ·  OPTION 2 — Rewriting tool schemas for different providers ✓ (keyed correct)  ·  OPTION 3 — Optimizing network latency  ·  OPTION 4 — Encrypting data before sending
Pass needs every checkpoint met — Question 9/9 · Options 8/8 · Answer-Validity 3/3 · Pedagogy 5/5. Any single miss → RED. The critic returns only the failed checkpoints; Python rolls them up.

🟢 GREEN Case 1 · clean → approve

Critic flags: problems: [] — nothing wrong.

Scoring: Question 9/9 · Options 8/8 · Answer-Validity 3/3 · Pedagogy 5/5 — every checkpoint met.

Outcome: band GREEN, passes → shown at the rubric gate ready to approve.

🔴 RED Case 2 · one minor miss → fail

Critic flags: just 2.1 — the option lengths are uneven. Everything else (incl. the answer key) is fine.

Scoring: Options 7/8 — one checkpoint short. Under the strict gate that alone fails the question.

Outcome: band RED → the optimize loop rewrites the options → re-scored (≤2 rounds). No "amber" — anything less than perfect is RED.

🔴 RED Case 3 · wrong answer caught

Setup: the key is wrong — it points at OPTION 1 (scaling infra), which the material never calls a manual task.

Critic flags: 3.1 — “the reading material supports OPTION 2 (rewriting tool schemas), not OPTION 1.”

Scoring: Answer-Validity 2/3 — a checkpoint unmet → band RED.

Outcome: optimize loop tries to fix; still red after 2 rounds → stated NOT passed → human gate. (Measured: the live critic caught 100% of wrong-answer keys across 4 sessions, 0% false alarms.)

⚠️ FAIL-SAFE Case 4 · critic can’t evaluate

Setup: the critic LLM times out or returns unparseable output for this batch.

What happens: the call is retried once; if it still fails, the question is flagged on 3.1 — “critic could not evaluate — flagged for human review.”

Outcome: band RED, NOT passed → human gate. Previously this path silently turned the whole batch green — that was the bug; now it fails safe.

♻️ LEARN Case 5 · reject → memory → next run

At the gate (after the loop): the human rejects Case 3 with a reason.

finalize writes memory: the reason is distilled into one generic, category-tagged rule, stored in feedback (deduped — a paraphrase bumps its hit-count & is kept as an alias, never deleted).

Next run: that rule is injected into the critic/optimize prompts; once it recurs enough it sharpens the checkpoint’s met_when itself. See §5.

🧠 Where memory touches the rubric (and where it doesn’t). READ during the loop — the rubric itself lives in memory (load_rubric), and learned feedback rules are injected as guidance into the critic/optimize prompts. WRITTEN right after the loop ends (finalize) — a reject/improve becomes a rule and, on recurrence, sharpens a checkpoint. But the met/not-met verdict is decided by the critic + deterministic Python (score_question) — never by memory. So when the critic can’t judge, the system states the question NOT passed rather than guessing.
4 Variants & export

Typed variants per question — in exact portal format

Each approved question is reformatted into variants that test the same outcome different ways. Every variant is emitted with the exact portal field keys, so the JSON + zip conversion is a direct mapping — no second transformation.

Per-base, language-aware (Feature 7). In a coding session, each approved base is classified: code-related bases get the code variant set (code-analysis MC + output-prediction, error-id, fix, functionality, logic, plus one free-text output and one fill-in-the-blank coding) authored in the session's detected language; conceptual bases keep the text set. A non-coding session is unchanged. Option-less types (CODE_ANALYSIS_TEXTUAL, FIB_CODING) export valid portal JSON with options: [] + an expected_output answer.

5 Three workflows, one engine

The same pipeline, pointed at three products

A workflow flag on the run picks the path; everything in between is reused unchanged. Classroom Quiz splits one session into Set 1 / Set 2 and grows typed variants. MCQ Practice takes the whole session as one pool and stops after the rubric — no variants. Module Quiz merges several sessions into one assessment, consolidates their outcomes, and keeps the full variants pipeline.

▥ Classroom Quiz

Split one session into Set 1 / Set 2 → plan → generate base MCQs → 3 gates (generation · rubric · variants) → each approved base grows 9 text + 8 code typed variants → export set_01 / set_02 + two portal zips.

◉ MCQ Practice Feature 8

No split — the whole session becomes one pool → plan (count always a multiple of 5) → generate questions spread across all content types → 2 gates (generation · rubric) → no variants → export a single mcq_practice quiz + one portal zip.

◈ Module Quiz Feature 9

Merge several selected sessions into one module body → an LLM picks the ≥6 most important concepts per session (no cap) → generate one assessment-level base per concept (APPLY / ANALYZE / EVALUATE — no rote recall) → 3 gates → the same typed variants → export a single module_quiz quiz + one portal zip.

The practice run — one continuous pass, two pauses

Same graph as the classroom flow with the two practice forks applied: no Split (the whole session is one pool) and no Variants / Gate 3 — it exports straight after the rubric gate. The self-evolving memory still feeds and learns from every step.

1 Whole-session plan

Reads the full reading material (no seam) and lists the learning outcomes — a count rounded to a multiple of 5 (5, 10, 15 …). One question per outcome.

2 Typed pool

Each question is authored as a content type — multiple choice, match, fill-in-the-blank, true/false, statement-based, multi-select, all/none, and code-analysis / FIB for coding sessions — validated by the same deterministic type checks the variants use.

3 Two human gates

Generation (accept · improve · delete the pool) then rubric (approve · improve · reject). The variants gate is skipped entirely.

4 Same learning

The 25-checkpoint rubric, the auto-optimize loop (≤2), and the self-evolving memory all apply — every reject / improve teaches, and learned rules are injected back into practice generation.

📦 Output. Approved questions are written to generated_quizzes/<course>/<session>/mcq_practice/ as mcq_practice.md + mcq_practice.json + reading_material.md, and packaged as {session}_mcq_practice.zip — one set-less pool, ready for the portal.
♻ Built by reuse, not duplication. Practice has its own nodes (generate_practice_pool, practice_gate, practice_export), prompts (16_mcq_practice_plan, 17_mcq_practice_generate) and pool label — so the word "set" never leaks in. The rubric scoring, both gates' learning loop, and the export writer are shared, made label-neutral.
5b A third workflow

Module Quiz — merge several sessions into one assessment Feature 9

Where the classroom flow splits one session, Module Quiz does the inverse: the reviewer multi-selects several sessions and the run merges them into one body. An LLM plans each session independently, picking that session's ≥6 most important concepts (no upper cap), then authors one assessment-level base per concept — APPLY / ANALYZE / EVALUATE with a scenario stem, never rote recall — then runs the same rubric + typed-variants pipeline as the classroom quiz. Code question types are added automatically when any selected session is a coding session.

1 Merge & select per session

The selected sessions are concatenated under per-session headers; an LLM (18_module_plan) runs once per session and picks that session's ≥6 most important, assessment-worthy concepts (no cap — more if the session is deep). The reviewer names the quiz (prefilled from the session titles) — that name is the export identity.

2 Assessment-level bases

One base per concept (19_module_generate) — every question is APPLY / ANALYZE / EVALUATE with a scenario stem, no rote recall or bare definitions. Count is uncapped (= sum of each session's concepts). Single module label, no Set 1 / Set 2.

3 Three human gates

Generation → rubric → variants — the full classroom review, rendered as one flat "module" column. Each approved base grows the same 9 text + 8 code typed variants.

4 Same engine, label-neutral

Reuses the rubric, variants, and export phases verbatim — the five per-set loops are now present_labels-driven, so one module label flows through unchanged. Every reject / improve still teaches the self-evolving memory.

📦 Output. Approved bases + variants are written to generated_quizzes/<course>/<module_slug>/module_quiz/ as module_quiz.md + module_quiz.json + reading_material.md, and packaged as {module_slug}_module_quiz.zip — one merged assessment, ready for the portal.
♻ Built by reuse, not duplication. Module Quiz adds only a merge-aware assemble, a generate_module node, and two prompts (18_module_plan, 19_module_generate). The rubric scoring, the typed-variants phase, both later gates' learning loop, and the export writer are all shared — finishing the label-neutrality the practice workflow began.
6 Self-evolving memory

It learns from every correction

Three small stores in Postgres. The one that makes it agentic is feedback: a human reject or improve becomes a generic rule, the rule is retrieved by relevance to the problem at hand, and a recurring rule sharpens the rubric itself. Nothing is ever deleted — rules evolve. Full deep-dive: .claude/plan/memory-and-feedback-explained.md.

1 Sessions

The ingested reading material (chunks + full source) the quiz is grounded in.

2 Rubric (global)

The rubric checkpoints — this is what the critic loads each run (load_rubric), editable, and auto-refined by the learning loop below.

3 Feedback (by category)

Generic authoring rules distilled from rejections & improvements — embedded for meaning-based retrieval and shown read-only at /learned (GET /api/feedback-rules).

How a rule is stored — evolve in place, never delete

a new rule arrives (distilled from a reject / improve)
  → same lesson already stored?  (cosine ≥ 0.85, same category — a paraphrase)
      YES → bump its hit_count + keep the new wording in aliases   (the lesson gets STRONGER — no new row)
      NO  → store a new embedded rule
  Nothing is ever deleted. `memory_admin fold` merges old duplicates by flagging them superseded (still stored).

How a rule is retrieved — by the PROBLEM, not the topic

# feedback_retrieval_mode = frequency | semantic | hybrid   (default: frequency)
generate · critic · variant-generate            → top rules by hit_count   (no failure to target — inject general guidance)
refine · optimize · variant-optimize · improve  → rules most relevant to the failure   (hybrid = relevance + hit_count)

e.g. problem "distractors are unrelated" → hybrid surfaces the distractor rules even at hit_count=1,
which a frequency-only top-8 would bury under unrelated high-count rules.

Where learned rules are injected (and on what basis)

Every author step pulls rules into its prompt's {{feedback_rules}} slot — not just Generate. Both helpers (rules_block(), _fb_block()) call one brain, relevant_feedback_rules(): a blank query (or frequency mode) → top-k by hit_count; a query in semantic/hybrid mode → vector search on that problem.

Step (prompt)CallQuery passedBasis
Generate (03)rules_block()— nonefrequency
Refine (05)rules_block(query=…)concept-check failure reasonsrelevance
Rubric critic (07)_fb_block()— none (judges all dimensions)frequency
Rubric optimize (08)_fb_block(query=…)failing-checkpoint reasonsrelevance

Only the rule text is injected (never hit_count/category/questions). Default mode is frequency, so out of the box every step is hit_count-ranked and the query= is ignored — turning on relevance is just memory_admin backfill + FEEDBACK_RETRIEVAL_MODE=hybrid.

The learning loop — reads before scoring, writes after the loop ends

human rejects or improves a question + gives feedback   (required at all 3 gates)
  → distill ONE generic, category-tagged rule   (no session facts — reusable everywhere)
  → store in feedback   (evolve-in-place: bump + alias, never deleted)
  → next run reads the relevant rules into the generate / critic / optimize prompts (guidance only)
  → when a rule recurs (hit-count ≥ threshold) → it sharpens that checkpoint's "met_when"   (tagged "learned from reviews" on the Rubric view)

Timing: guidance is READ each run BEFORE scoring; corrections are WRITTEN after the loop ends (finalize).
The verdict itself stays with the critic + Python — memory shapes what the agent applies tomorrow, not today's math.
7 Cost, speed & observability

Fast, measured in rupees, fully traceable

⚡ Faster by design

Independent model calls run on a bounded thread pool; round-2 re-scores only what changed; telemetry writes are batched off the critical path. Typically ~65% less model wall-time (≈12 min → ≈4 min).

💰 Real cost in ₹ & $

The run snapshots OpenRouter usage before/after, so you see the exact rupee cost of that session (e.g. "₹25.18 ≈ $0.302"), plus a live account-balance badge. Rate is configurable (USD_TO_INR).

🔭 Per-session traces

With LANGSMITH_TRACING=true + a key in .env, every run is a full LangSmith trace tree — per-call latency, tokens and cost — the drill-down view for any individual session.

📁 Two manager views, by intent. This document is the whole-system architecture (the demo). For one specific session, open its run in LangSmith to see every prompt, gate decision, token and rupee — no per-session HTML to dig through.

1 The pipeline

One run: extract → harvest → verify → (review) → publish

A single LangGraph run, checkpointed to Postgres, forks on workflow == "mock_interview". Well-evidenced questions auto-publish; only the under-evidenced few reach a human — so intervention trends toward zero.

Defining principle. The LLM only (1) names the outcomes, (2) selects genuine questions from candidates pulled verbatim by the source connectors, and (3) confirms which companies the evidence supports. It never authors a question or invents a company — that's what makes "real interview questions" actually real.
2 How it fetches

Two ways in — then one shared extraction step

It does not visit all 94 sources one by one. There are two retrieval styles, and then the same step pulls out question · company · link, verbatim.

📖 Read the page directly

For sources that allow it: the GitHub repos (GitHub API) and the article/blog sites (polite fetch → trafilatura clean text). These are exact, known addresses we read.

🔦 Search via Tavily — a trusted-site filter

For the rest (Glassdoor, Blind, AmbitionBox, LeetCode, listicles…): Tavily searches the web per outcome, and we keep only results whose domain is on the allowlist. The allowlist is a filter, not a visit-every-site list — so adding domains widens coverage cheaply.

Shared last step. Whatever text comes back, the model extracts exactly three things — question · company name · link — taking them word-for-word. No company is ever guessed.
3 The evidence engine

Cluster → corroborate → strict bar → research → ground

This is what turns scattered hits into "asked at N companies, here are the links." It runs inside verify, the agentic core.

🧩 1 · Cluster

Embed every harvested question (text-embedding-3-small) and group near-duplicates (cosine ≥ 0.84). The same question worded differently across sources becomes one row.

🏢 2 · Corroborate

Across a cluster, collect the distinct companies (each with a proof URL) and all source domains. The company count is real evidence found — never padded to a target.

🔁 3 · Research loop

If a question is real but under-evidenced (one source, or no clear company), re-search Tavily for stronger proof — up to 3 rounds. Adaptive, where it actually pays.

🔗 4 · Link-resolve

Every cited link is checked. An anti-bot 401/403/429 still means the page exists; only 404/410/dead fails and is dropped.

✓ 5 · Ground (prompt 22)

For each question the model re-reads the evidence and reports only the companies the snippets actually support — dropping any hallucinated attribution. Steered by the self-evolving skill.

⚖️ 6 · The strict bar

Publish only with > 1 independent live domain + a grounded company + resolving links. Everything else is queued or dropped.

4 Review & learning

Publish · Queue · Drop — and a skill that sharpens itself

Every clustered question lands in one of three buckets. Only the middle bucket asks a human, and every rejection makes the next run smarter.

PUBLISH

Passes the strict bar → auto-published to the bank. No human needed.

QUEUE

A genuine, on-topic question that's real but under-evidenced → the exception-only human queue. (The gate is skipped entirely when this bucket is empty.)

DROP

Not a question, fabricated, off-topic, or no live source → discarded.

The skill self-evolves. The verifier's criteria live in a repo file, .claude/skills/verifying-interview-evidence/SKILL.md — not a database. When a reviewer rejects a queued row with a reason, it's distilled into a one-line rule and appended to the skill's "Learned rules" section (git-versioned). Next run, those rules steer the grounding step — so fewer weak candidates ever reach a human.
# SKILL.md → after a reviewer rejects "sourced only from one training blog" ## Learned rules - A single training-blog listicle is not an independent source. - Require a named company tied to the question in the snippet, not just the page topic.

6 Cost

A few cents of LLM + a handful of search credits

Exact per-run cost is captured the same way as every other workflow, plus the one net-new line item (search credits).

🤖 LLM (OpenRouter)

Extraction, harvest-filter, grounding + the inline distiller run on claude-haiku-4-5 / the Sonnet critic. The driver reads OpenRouter's cumulative usage before/after the run and stores the exact ₹/$ diff — surfaced as cost_usd / cost_inr.

🔦 Tavily search credits

~1 query per outcome + the bounded research loop ⇒ roughly 15–60 searches/run. This is the only cost outside the OpenRouter meter; it's small and tracked in your Tavily dashboard.

Order of magnitude: a topic run is typically well under $1 of LLM spend plus a few dozen Tavily credits — and re-runs get cheaper as the bank dedupes and the skill prunes weak candidates.

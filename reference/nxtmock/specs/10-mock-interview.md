# 10 — Mock Interview (harvest REAL, evidenced interview questions)

> The contract for a new product type, `mock_interview`. This document says **exactly what to build**.
> It is a **harvest-and-verify** workflow: questions are pulled from real sources and proven — **never
> generated by the LLM**. Companion build steps are in `.claude/plan/feature-11-mock-interview.md`.

## A. Principle (non-negotiable)

The LLM **never authors a question and never invents a company**. Its only jobs are: (1) extract the
selected sessions' **key learning outcomes/skills**, (2) **find** real interview questions tied to those
outcomes that were asked at real companies, (3) **extract** them verbatim with their evidence, and
(4) **verify** that evidence is real. Every published question must carry **more than one independent
legitimate source + at least one named company + links that resolve**. The company count reflects only
real proof found — it is **never padded** to a target.

## B. Product placement

A **separate, on-demand workflow** alongside Classroom Quiz / MCQ Practice / Module Quiz. It reuses the
existing engine (session merge, the ≤3-round loop, gate/resume lifecycle, SSE, cost, frontend pickers)
but is implemented in its own module and leaves the other workflows untouched. **No daily scheduler.**
The deliverable is a **Markdown file**, not portal json/zip.

| | Mock Interview |
|---|---|
| Input | several sessions of a topic, **merged** (reuse `module_quiz`'s multi-select + merge) |
| Source of questions | **real interview sources on the web** (see §F), not the LLM |
| Gen node(s) | `extract_outcomes` → `harvest` → `verify ⟲ research` (new `backend/agent/mock_interview.py`) |
| Evaluation | a repo **`SKILL.md`** (not a DB rubric) — evidence checks, self-evolving |
| Human gates | **one, exception-only** — reviews just the unresolved/conflicting residue |
| Output dir | `generated_quizzes/{course}/{topic_slug}/mock_interview/` |
| Output files | `mock_interview.md` (deliverable) + `mock_interview.json` (raw rows, for re-load) |
| Models | extraction/verification on `anthropic/claude-haiku-4-5`; embeddings `text-embedding-3-small` |

## C. End-to-end flow

One LangGraph run, checkpointed to Postgres, forking on `state["workflow"] == "mock_interview"`.

```
START
  → assemble          merge the selected sessions (reuse nodes._load_module_content)
  → extract_outcomes  [Agent 1] pull the key outcomes/skills from the merged body
  → harvest           [Agent 2] for each outcome, fan out across ALL enabled source connectors (§F)
                                via pmap; EXTRACT-ONLY → {question, company, url, snippet} as literally found
  → verify ⟲ research [Agent 3] per candidate, INDEPENDENTLY, guided by SKILL.md (§E):
                                · each url resolves (HTTP 200)
                                · the page actually contains that question AND that company
                                · cross-source corroboration → count DISTINCT companies
                                weak/unmet ⇒ re-search for better proof   [⟲ ≤ 3 rounds]
  → publish/queue     corroborated (passes the bar) ⇒ status=published (auto, no human)
                      unresolved/conflicting        ⇒ status=queued  (small human queue)
  → interview_gate    👤 reviewer sees ONLY queued rows: approve / edit / reject (reason required)
  → export_md         write mock_interview.md (+ .json); append distilled reject reasons to the
                      SKILL.md "Learned rules" section (the skill self-evolves)
  → END
```

`harvest` and `verify` are the analogue of `process_set`'s `generate` and `concept_check ⟲ refine`. The
verify loop reuses the `MAX_REFINE_ROUNDS = 3` budget from `domain/state.py`.

## D. The three agents (what each must do)

1. **`extract_outcomes`** — input the merged session body; output the deduped list of key
   outcomes/skills the topic teaches. Reuse the planning prompt pattern (`02_plan_outcomes.md` /
   `18_module_plan.md`). One new prompt: `20_interview_outcomes.md`.
2. **`harvest`** — input the outcomes; for each, query every enabled connector (§F) and return raw
   candidate records **exactly as found** — no inference, no paraphrasing of company names. One new
   prompt `21_interview_harvest.md` governs extraction from fetched page text (pull the literal question
   string + any named company + the URL it came from). Fan out with `concurrency.pmap`.
3. **`verify`** — input the candidates; for each, independently confirm evidence against the live
   sources and the `SKILL.md` checks, corroborate across sources, count distinct companies, and decide
   `published` vs `queued` vs `dropped`. May re-search (bounded ≤3 rounds) for stronger proof. One new
   prompt `22_interview_verify.md`. The verdict is computed by an explicit Python gate (`evidence_gate`,
   §E) — the LLM supplies the grounded judgements, Python applies the binary rule.

## E. Evaluation — `SKILL.md`, self-evolving (NOT a DB rubric)

Create a repo skill at `.claude/skills/verifying-interview-evidence/SKILL.md`. It defines the checks the
`verify` agent applies. There is **no** rubric in PostgresStore for this workflow.

**Hard checks (all must pass to publish):**
1. **Real, not fabricated** — the question text is literally present on a fetched source page.
2. **>1 independent source** — corroborated by **more than one** independent legitimate source
   (independent = different domains/authors, not two pages of one site).
3. **Named company + resolving links** — at least one explicitly named company, and **every** cited URL
   returns HTTP 200 and actually contains the claim.
4. **Outcome-relevant** — maps (by embedding similarity + judgement) to one of the selected outcomes.
5. **Not a duplicate** — below the dedup similarity threshold vs the published bank.
6. **Safe/appropriate** — no PII, not offensive, genuinely an interview question.

**`evidence_gate(candidate) -> {decision, reason}`** (deterministic Python): `published` if all hard
checks pass; `queued` if real-but-borderline (e.g. exactly one strong source after the loop, or a
company named without a resolving link); `dropped` if fabricated/irrelevant/unsafe.

**Company-tagged-page rule (Part 7).** A company parsed directly from a **company-tagged source URL**
(AmbitionBox `/interviews/<company>-…`, Glassdoor `/Interview/<company>-…`, Exponent `/guides/<company>-…`,
Levels `/companies/<company>/`) is treated as **authoritative proof** — the page *is* that company's
interview page, so its URL is the evidence. Such a question **publishes even from a single live source**
(it relaxes hard-check #2's ">1 source" for this case only). The research loop runs a company-scoped
search pass (`include_domains` over the attribution sites) so these pages surface for each question. All
other (snippet/title-derived) company hints still go through strict prompt-22 grounding; no-company /
no-live-source questions still queue or drop. This is the recall fix so real companies stop showing "—".

**Self-evolution:** on reviewer reject, `feedback.distill_and_persist` produces a one-line generic rule;
the workflow **appends it to the SKILL.md "Learned rules" section** (git-versioned). Subsequent runs feed
those learned rules into `21`/`22`, so the verifier gets stricter and **fewer candidates reach the human
over time**. The persistence target is the **skill file**, not a memory namespace.

## F. Sourcing — connectors + the ≥50-source catalog (the core)

A pluggable connector layer. Each connector implements `fetch(outcomes) -> list[record]` where a record
is `{question_text, company|None, source_url, raw_snippet, source_type}`. All connectors are individually
toggleable in `.env`.

### F.1 How the 90+ sources are wired (AS BUILT — not one scraper per site)
**Two enabled connectors** (`backend/agent/sources/`), both toggleable in `.env`:
- **GitHub repos connector** (`github_repo.py`, GitHub API; license-clean) — reads the repos in
  `interview_github_repos` (catalog group C). `owner/repo` + file paths are regex-validated; the GitHub
  API path uses the SSRF-guarded `polite_get`.
- **Tavily search connector** (`tavily_search.py`) — searches the whole `interview_source_allowlist` and
  returns extracted content, keeping only results whose domain is on the allowlist. **This one connector
  reaches every allowlisted site** (GeeksforGeeks, AmbitionBox, Glassdoor, Levels, Blind, **Reddit**, the
  listicles, …) **legitimately via search** — so there is NO per-site scraper, and no ToS-violating direct
  scraping of anti-bot sites. The research loop reuses it (`search_question`).

> **Deviation from the original sketch (recorded in `feature-11` Part 2):** dedicated GeeksforGeeks /
> Reddit / AmbitionBox connectors are **not** built — Tavily covers them via the allowlist. They remain a
> future option (e.g. a Reddit official-API connector) if a source ever needs first-class extraction.
> `enabled_connectors()` is the registry; add a connector there behind a settings flag.

### F.3 Tooling, ethics & security (as built)
**Extraction:** Tavily returns extracted content itself; `trafilatura` is available for any direct
`polite_get` fetch. (`crawl4ai` is **not** a dependency — Tavily's extraction suffices; deferred.)
**Legal/ethics:** prefer API/license-clean sources; respect `robots.txt`, rate-limit, descriptive UA;
**store the question + a link, never bulk-republish article bodies**; per-source kill switch in `.env`.
**Security guards (added after a security review):**
- **SSRF:** `is_safe_public_url()` blocks non-http(s) + private/loopback/link-local/metadata hosts before
  any fetch; the link-resolve check uses `follow_redirects=False`; the fan-out is capped
  (`_MAX_LIVE_CHECK_URLS`).
- **Prompt injection:** scraped question/snippet text is collapsed to a single bounded line (`_one_line`)
  before prompts 21/22; the structured-output schemas + verbatim/grounding rules are the primary defense.
- **Learning safety:** reviewer reason is length-bounded; the distilled rule is validated (no `##`/`<!--`,
  ≤200 chars) and capped per run (`_MAX_RULES_PER_RUN`) before being written to `SKILL.md`.
- **Secrets:** connector exceptions are never echoed into events (could carry an auth header); `owner/repo`
  + GitHub file paths are regex-validated.

## G. Data model

- **Bank** — PostgresStore namespace `("interview_bank", course, topic_slug)`, **indexed on
  `question_text`** (embedding dedup). Item fields:
  `question_text, outcome, companies: [{name, url}], company_count, source_urls: [str],
  status: "published"|"queued"|"rejected", first_seen, last_seen, run_ids: [str]`.
  Re-running a topic **merges duplicates** (embedding match) and **grows** companies/sources rather than
  inserting a new row. `company_count = len(distinct companies)`.
- **No `interview_rubric` namespace.** Evaluation lives in the skill file (§E).

## H. Output (the deliverable)

`generated_quizzes/{course}/{topic_slug}/mock_interview/mock_interview.md` — a GitHub-flavored Markdown
table, rows ordered by `# Companies` descending:

```
| # | Interview Question | Outcome / Skill | Companies | # Companies | Source links |
```

- **Companies** — each company name rendered as a Markdown link to the proof page for that company.
- **Source links** — all corroborating URLs (≥2).
- A sibling `mock_interview.json` holds the raw rows for re-load; the **.md is the product**.

## I. Settings / keys (`backend/settings.py` + `.env`)

`tavily_api_key`, `interview_source_allowlist: list[str]`, `interview_github_repos: list[str]`,
`interview_dedup_similarity: float = 0.88`, per-source enable flags
(`interview_source_<name>_enabled: bool`), optional `reddit_client_id`/`reddit_client_secret`,
optional `github_token`. New deps: `tavily-python`, `crawl4ai`, `trafilatura`, `httpx`
(and `PyGithub` or raw `httpx` for the GitHub connector).

## J. Acceptance criteria

1. A run on selected sessions emits `mock_interview.md` where **every** row has ≥1 named company, ≥2
   resolving corroborating links, and maps to a session outcome.
2. **Anti-hallucination:** a fabricated/404 source or a question absent from its page is dropped; a
   single-source question is held (`queued`), never padded into the table.
3. The human gate surfaces **only** `queued` rows (auto-published ones never reach it).
4. Rejecting a queued row with a reason **appends a rule to the SKILL.md "Learned rules"**, and a re-run
   applies it (fewer rows queued) while **merging duplicates / growing company_count**.
5. Existing workflows (classroom/practice/module) are byte-for-byte unaffected.

> **Delivery:** built in **3 runnable parts** — P1 walking skeleton (GitHub source only, relaxed bar,
> auto-publish) → P2 all 60 sources + the strict verify/evidence engine → P3 human queue + self-evolving
> `SKILL.md`. Each part is independently demoable. Acceptance criteria 1–4 above are fully met at the end
> of **P3**; P1/P2 have their own reduced DoD (see `.claude/plan/feature-11-mock-interview.md`).
